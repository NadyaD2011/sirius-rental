## 🌐 Общая архитектура

Проект «Сириус.Аренда» построен по архитектуре **клиент-сервер** с разделением на фронтенд и бэкенд. Бэкенд реализован как REST API, фронтенд — как статическое веб-приложение.

```

│  │ Веб-браузер  │  │  Swagger UI  │  │  Postman/cURL    │  │
│  │ (HTML/CSS/JS)│  │  (/docs)     │  │                  │  │
│  └──────┬───────  └──────┬───────┘  └────────┬─────────┘  │
└─────────┼─────────────────┼───────────────────┼────────────
```

## 🎯 Архитектурный стиль

### REST API

Проект следует принципам **REST (Representational State Transfer)**:

| Принцип | Реализация |
|---------|-----------|
| **Клиент-сервер** | Разделение фронтенда и бэкенда |
| **Stateless** | JWT-токены не хранят состояние на сервере |
| **Единообразие интерфейса** | Стандартные HTTP-методы (GET, POST, PUT, DELETE) |
| **Ресурсы** | `/users`, `/rooms`, `/bookings` |
| **Слои** | Клиент → API → Бизнес-логика → БД |

### HTTP-методы

| Метод | Назначение | Пример |
|-------|-----------|--------|
| `GET` | Получение данных | `GET /rooms/` |
| `POST` | Создание ресурса | `POST /bookings/` |
| `PUT` | Обновление ресурса | `PUT /rooms/{id}` |
| `DELETE` | Удаление ресурса | `DELETE /bookings/{id}` |

##  Компоненты системы

### 1. Фронтенд (Клиентская часть)

**Технологии:** HTML5, CSS3, JavaScript (ES6+), FullCalendar.js

**Компоненты:**
- **Страница авторизации** — вход, регистрация, восстановление пароля
- **Главная страница** — список комнат, поиск, бронирование
- **Админ-панель** — управление пользователями, комнатами, статистика
- **Календарь** — визуализация расписания бронирований

**Особенности:**
- Адаптивный дизайн (мобильные устройства)
- Тёмная/светлая тема
- Single Page Application (SPA) поведение без фреймворков

### 2. Бэкенд (Серверная часть)

**Технологии:** Python 3.11, FastAPI, SQLAlchemy, Pydantic, JWT

**Компоненты:**
- **API Layer** — маршрутизация запросов (`routers/`)
- **Business Logic** — бизнес-правила (`auth.py`, проверка пересечений)
- **Data Access Layer** — работа с БД (`models.py`, `database.py`)
- **Validation** — валидация данных (`schemas.py`)

### 3. База данных

**Технологии:** PostgreSQL (продакшн), SQLite (разработка)

**Особенности:**
- Реляционная модель
- Внешние ключи для целостности данных
- Поддержка транзакций

## 🗄️ Структура базы данных

### ER-диаграмма

```
┌─────────────────┐       ┌──────────────────┐       ┌─────────────────┐
│     users       │       │    bookings      │       │     rooms       │
├─────────────────┤       ├──────────────────┤       ├─────────────────┤
│ id (PK)         │◄──────│ user_id (FK)     │       │ id (PK)         │
│ username        │       │ room_id (FK)     │──────►│ name            │
│ hashed_password │       │ start_time       │       │ capacity        │
│ role            │       │ end_time         │       │ equipment       │
└─────────────────┘       │ status           │       └─────────────────┘
                          └──────────────────┘
```

### Описание таблиц

#### Таблица `users`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer (PK) | Уникальный идентификатор |
| `username` | String (Unique) | Имя пользователя |
| `hashed_password` | String | Хэш пароля (bcrypt) |
| `role` | String | Роль: `user` или `admin` |

#### Таблица `rooms`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer (PK) | Уникальный идентификатор |
| `name` | String | Название пространства |
| `capacity` | Integer | Вместимость (человек) |
| `equipment` | String | Оборудование (список через запятую) |

#### Таблица `bookings`
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer (PK) | Уникальный идентификатор |
| `room_id` | Integer (FK) | Ссылка на комнату |
| `user_id` | Integer (FK) | Ссылка на пользователя |
| `start_time` | DateTime | Время начала |
| `end_time` | DateTime | Время окончания |
| `status` | String | Статус: `active` или `cancelled` |

### Связи

- **One-to-Many:** `users` → `bookings` (один пользователь может иметь много бронирований)
- **One-to-Many:** `rooms` → `bookings` (одна комната может иметь много бронирований)

## 🏗️ Слои приложения

### 1. Presentation Layer (API Layer)

**Файлы:** `app/routers/*.py`

**Ответственность:**
- Приём HTTP-запросов
- Валидация входных данных (Pydantic)
- Возврат HTTP-ответов с правильными статус-кодами

**Пример:**
```python
@router.post("/bookings/", response_model=BookingOut, status_code=201)
def create_booking(
    booking: BookingCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # Делегирование бизнес-логике
    ...
```

### 2. Business Logic Layer

**Файлы:** `app/auth.py`, логика проверки пересечений

**Ответственность:**
- Аутентификация и авторизация
- Проверка бизнес-правил (пересечения бронирований)
- Ролевая модель

**Пример:**
```python
def check_booking_conflict(db, room_id, start_time, end_time):
    """Проверка пересечения бронирований"""
    return db.query(Booking).filter(
        Booking.room_id == room_id,
        Booking.status == "active",
        Booking.start_time < end_time,
        Booking.end_time > start_time
    ).first() is not None
```

### 3. Data Access Layer

**Файлы:** `app/models.py`, `app/database.py`

**Ответственность:**
- Взаимодействие с базой данных
- ORM-модели (SQLAlchemy)
- Управление сессиями БД

**Пример:**
```python
class Room(Base):
    __tablename__ = "rooms"
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)
    capacity = Column(Integer, nullable=False)
    equipment = Column(String, default="")
```

### 4. Validation Layer

**Файлы:** `app/schemas.py`

**Ответственность:**
- Валидация входных данных
- Сериализация/десериализация JSON
- Документация API (автоматическая через Pydantic)

**Пример:**
```python
class BookingCreate(BaseModel):
    room_id: int
    start_time: datetime
    end_time: datetime
```

## 🎨 Паттерны проектирования

### 1. Dependency Injection

**Использование:** FastAPI `Depends()`

**Цель:** Внедрение зависимостей (сессия БД, текущий пользователь)

**Пример:**
```python
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@router.get("/rooms/")
def get_rooms(db: Session = Depends(get_db)):
    ...
```

### 2. Repository Pattern (через ORM)

**Использование:** SQLAlchemy ORM

**Цель:** Абстракция доступа к данным

**Пример:**
```python
# Вместо SQL-запросов используем ORM
rooms = db.query(Room).filter(Room.capacity >= 10).all()
```

### 3. Middleware Pattern

**Использование:** CORS middleware

**Цель:** Обработка кросс-доменных запросов

**Пример:**
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### 4. Singleton (для engine БД)

**Использование:** `engine` в `database.py`

**Цель:** Один экземпляр подключения к БД на всё приложение

## 🔐 Безопасность

### Аутентификация

**Механизм:** JWT (JSON Web Tokens)

**Процесс:**
1. Пользователь отправляет логин/пароль → `POST /users/login`
2. Сервер проверяет пароль (bcrypt) → генерирует JWT-токен
3. Клиент сохраняет токен (localStorage)
4. Каждый запрос включает токен: `Authorization: Bearer <token>`
5. Сервер декодирует токен → извлекает `username` → находит пользователя

**Структура токена:**
```json
{
  "sub": "username",
  "role": "user",
  "exp": 1720857600
}
```

### Авторизация

**Ролевая модель:**
- `user` — базовые права (бронирование, просмотр)
- `admin` — расширенные права (CRUD комнат, управление пользователями)

**Реализация:**
```python
def require_admin(current_user: User = Depends(get_current_user)) -> User:
    if current_user.role != "admin":
        raise HTTPException(status_code=403, detail="Требуется роль администратора")
    return current_user
```

### Хеширование паролей

**Алгоритм:** bcrypt

**Особенности:**
- Соль генерируется автоматически
- Стойкость к brute-force атакам
- Невозможность восстановления пароля из хэша

### CORS

**Настройка:** Разрешены запросы с любых источников (для разработки)

**Продакшн:** Следует ограничить `allow_origins` конкретными доменами

## 🐳 Развертывание

### Локальное развертывание

```bash
# 1. Установка зависимостей
pip install -r requirements.txt

# 2. Инициализация БД
python init_db.py

# 3. Запуск сервера
uvicorn app.main:app --reload
```

### Docker-развертывание

**Архитектура контейнеров:**
```
┌─────────────────────────────────────────┐
│           Docker Compose                │
│                                         │
│  ┌──────────────┐    ┌──────────────┐  │
│  │   backend    │───►│      db      │  │
│  │  (FastAPI)   │    │ (PostgreSQL) │  │
│  │   :8000      │    │   :5432      │  │
│  └──────────────    └──────────────┘  │
└─────────────────────────────────────────┘
```

**docker-compose.yml:**
```yaml
services:
  db:
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
  
  backend:
    build: .
    depends_on:
      db:
        condition: service_healthy
```

**Преимущества:**
- Изоляция зависимостей
- Воспроизводимость окружения
- Простота масштабирования

## 📈 Масштабируемость

### Горизонтальное масштабирование

**Бэкенд:**
- Stateless-архитектура (JWT не хранит состояние)
- Возможность запуска нескольких инстансов за load balancer

**База данных:**
- PostgreSQL поддерживает репликацию
- Read replicas для чтения, master для записи

### Вертикальное масштабирование

**Оптимизации:**
- Индексы на часто используемых полях (`room_id`, `user_id`, `status`)
- Кэширование частых запросов (Redis — будущее улучшение)
- Асинхронные операции (FastAPI поддерживает `async/await`)

### Будущие улучшения

1. **Кэширование:** Redis для хранения популярных запросов
2. **Очереди задач:** Celery для фоновых операций (email-уведомления)
3. **Микросервисы:** Разделение на сервисы пользователей, бронирований, уведомлений
4. **GraphQL:** Альтернатива REST для гибких запросов
5. **WebSocket:** Real-time обновления расписания
